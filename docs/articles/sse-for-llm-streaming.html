BACKEND · STREAMING

用 SSE 把工作流进度和 LLM 输出送到前端

对“服务端持续推送、客户端主要接收”的场景，SSE 是比双工协议更轻的选择：浏览器保持一个 HTTP 长连接，服务端按事件发送增量数据。

学习笔记 · 依据本地《检索 Web 服务端搭建》整理

什么时候适合 SSE

导入文档时，用户需要知道当前正在解析、切片还是向量化；模型回答时，用户希望看到文字逐步出现；后台任务运行时，前端需要获得错误和完成信号。这些都是服务端单向、持续输出状态的场景。若客户端需要高频发送控制消息或实时协作，再考虑双工通信。

事件设计

event: progress data: {"task_id":"…","stage":"chunking","percent":42} event: answer data: {"task_id":"…","delta":"正在生成…"} event: done data: {"task_id":"…"}

把 progress、answer、error 和 done 分开，前端才能分别更新步骤条、正文、错误提示和按钮状态。每条消息都带任务 ID，可以避免并发请求的输出混在一起。

边界与恢复

浏览器自动重连并不代表任务仍然存在。服务端需要定义重连后能否继续、任务结束后何时关闭连接、错误如何转换成用户可读的信息。对于长任务，也要控制缓存和重放范围，避免每次重连都重复发送全部日志。
