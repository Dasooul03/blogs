RAG · FOUNDATIONS

从“先查资料，再回答”理解 RAG

RAG 的重点不是“让模型记住更多”，而是在回答前为它找到能引用、能更新、也能被核对的上下文。

学习笔记 · 2026 / 07 · 预计阅读 6 分钟

为什么不把全部文档直接塞进 Prompt？

首先是容量和成本：企业文档或长期笔记往往远超上下文窗口；即便装得下，无关内容也会占用 token，并让模型更难找到真正相关的信息。更重要的是，回答无法自然追溯到原始依据。

我对 RAG 的一句话理解：它只把“此刻最相关的证据”送进模型的上下文，而不是要求模型凭记忆回答。

两条链路，而非一次调用

离线索引：文档 → 提取文本 → 按语义切片 → Embedding → 向量库 在线回答：问题 → Query Embedding → Top-K 检索 → 组装上下文 → LLM 回答

把这两个阶段分开，学习时会更容易定位问题：答案差，可能是文档没有被正确解析；也可能是切片边界不对、召回不全、排序有噪音，或者最终提示词没有约束模型只依据证据作答。

四个最先该实验的变量

变量需要观察什么
切片大小与重叠语义是否被切断；一块里是否混入过多主题。
Top-K召回数量增加后，答案是更完整还是更嘈杂。
元数据能否保留标题、来源与章节，以便展示引用和过滤。
无答案策略证据不足时能否明确说“不知道”，而不是补全猜测。

下一步

先做一个只支持 Markdown 的最小知识库：准备 10 个有标准答案的问题，记录命中的片段和最终回答。确认“检索”可测以后，再加入 PDF、重排序与复杂工作流。
