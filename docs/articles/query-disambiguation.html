RAG · QUERY UNDERSTANDING

检索前，先确认用户到底在问什么

当一个知识库包含多个产品、多个版本的资料时，直接检索可能把问题带到错误的文档集合。先做对象确认和指代消歧，往往比多搜几次更重要。

学习笔记 · 依据本地《检索数据节点实现与测试》整理

一个确认节点做四件事

  1. 读取有限轮的历史会话，补足“它”“这个”等指代。
  2. 保存当前问题，让对话和排错可以追溯。
  3. 提取候选对象并重写问题，输出稳定的结构化结果。
  4. 将候选映射到知识库标签；低置信度时请用户确认。

置信度不是装饰

对象匹配可以有高、中、低三个结果。高置信度时进入对应资料范围；中置信度时给出候选项和当前理解；低置信度时停止检索并追问。把这条规则写清楚，比让模型在不确定时继续回答更能避免“听起来合理但查错资料”的结果。

重写问题的边界

问题改写的目标是消除指代和补足会话上下文，不是改变用户意图。记录原问题、重写问题和候选对象，便于复盘改写是否越界。对于多对象问题，宁可拆成多个查询或请用户选择，也不要把多个对象混成一个检索向量。

测试样本

至少准备明确对象、代词追问、对象不存在、多个对象并列四类问题。检查每类问题进入的知识库范围是否正确，再评估最终回答。这样能把“检索不准”拆成对象确认、召回和生成三个可分别修复的问题。
