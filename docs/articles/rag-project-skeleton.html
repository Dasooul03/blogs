PDF / Markdown → 结构化解析 → 语义切片 + 元数据 → 稠密/稀疏向量 → 向量库

骨架里需要的四个责任边界

模块职责常见失败
解析保留标题层级，让图片和表格不脱离段落。只得到无结构的纯文本。
切片形成可独立理解的知识片段。片段太碎或混入多个主题。
嵌入生成检索信号，同时保留原文和元数据。只存向量，丢失来源与调试信息。
持久化保存索引、源文件和任务记录。无法重建或定位一次导入。

先让数据流有固定出口

每次导入都应产生可定位的任务记录：原文件在哪里、解析后的文本在哪里、切片和向量由哪个版本生成。替换解析器或调整切片策略时，就能只重跑受影响的步骤，不必把所有资料从头处理。这个要求会直接影响目录结构、命名和日志字段。

依赖分层的实际意义

解析 PDF、调用 Embedding、连接向量库和提供 Web 接口的依赖变化频率不同。把它们放在适配层，工作流只依赖统一接口，某个 SDK 改动时不会牵动整个项目。日志则至少记录任务 ID、步骤名、输入摘要、耗时和错误原因，而不是只打印一段异常文本。

最小落地顺序

先实现 Markdown 导入、固定切片和本地检索，再加入 PDF、图片和稀疏向量。每加一层都保留一组回归问题，确认新能力没有让旧资料的结果退化。
