BACKEND · FASTAPI

从一个最小 FastAPI 开始

Agent 从脚本走向服务，第一步不必复杂：一个 GET 路由、一个 POST 路由和一份明确的请求模型，就已经能把能力交给前端或其他服务调用。

学习笔记 · 依据本地《Phase 0 Day 2：FastAPI 最小 API》整理

最小 API 的三件事

概念作用在 Agent 服务中的例子
路由定义路径与 HTTP 方法。/health、/ask、/tasks/{id}
请求模型约束输入字段和类型。问题、会话 ID、可选的知识库范围。
响应模型告诉调用方会得到什么。答案、来源、任务状态和错误信息。

为什么数据校验要放在入口

如果问题为空、会话 ID 格式不对或请求字段缺失，应在接口入口就返回清楚的错误。这样后面的检索和模型节点可以假定输入可用，日志也更容易区分“用户输入不合法”和“服务内部失败”。

从同步到异步任务

短问题可以直接返回答案；文档导入、批量 Embedding 这类耗时操作则更适合返回任务 ID，再用查询接口或 SSE 展示状态。把两种模式分开，前端不会因为等待一个长任务而失去响应。
