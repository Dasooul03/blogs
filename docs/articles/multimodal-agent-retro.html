AGENT · ENGINEERING RETROSPECTIVE

多模态 Agent 的一次工程复盘

把语音、画面与模型调用串起来以后，真正难的往往不是“模型能不能回答”，而是系统在网络波动、输入不同步和成本约束下还能不能稳定工作。

工程复盘 · 已脱敏整理 · 2026 / 07

从实时链路退回可靠链路

最初的实时通信方案在超时、重连和服务端错误上不够稳定。复盘后选择更简单的 HTTP 管线：单次响应的等待可能略长，但失败边界更清楚，也更容易观察、重试和降级。这个取舍提醒我，原型期应先保证一次任务可以完整结束，再追求理想的实时体验。

输入如何汇合

语音与画面两条输入管线并行。模型调用前读取一个短时间窗口内最近可用的画面；如果画面缺失则转为纯文本模式。这样不会因为某一路输入晚到就让整个会话失败，同时也避免拿到过期画面后仍被当作当前上下文。

四个工程决策

问题处理方式
无效调用与成本按需截图、压缩图像、过滤静音、限制历史轮次，并为不同任务拆分模型选择。
上游失败为视觉输入设置超时；有缓存则使用缓存，没有则明确走文本模式。
模块耦合用事件总线衔接语音识别、视觉、推理和语音合成，避免模块直接互相依赖。
过期任务后续应加入任务版本或 generation counter，避免旧结果覆盖新意图。

下一次会先补什么

先补生成过程的流式反馈和用户打断能力，再考虑长期记忆。前两项决定一次交互是否自然、是否可控；只有链路稳定后，向量化摘要才值得进入系统，否则它会把不稳定体验永久保存下来。
