AGENT · ENGINEERING RETROSPECTIVE

多模态 Agent 的一次工程复盘

把语音、画面与模型调用串起来以后，真正难的往往不是“模型能不能回答”，而是系统在网络波动、输入不同步和成本约束下还能不能稳定工作。

工程复盘 · 已脱敏整理 · 2026 / 07 · 预计阅读 6 分钟

从实时链路退回可靠链路

最初的实时通信方案在超时、重连和服务端错误上不够稳定。复盘后选择更简单的 HTTP 管线：虽然单次响应增加了一点等待，但换来了更可预测的失败处理与更容易定位的问题边界。

在原型阶段，可靠的“能完成一次任务”通常比脆弱的“看起来实时”更有价值。

四个关键决策

问题处理方式
音画不同步两条输入管线并行；Agent 调用前读取短时间窗口内的最近画面，缺失时降级为纯文本。
无效调用与成本按需截图、压缩图像、过滤静音、限制历史轮次，并为不同任务拆分模型选择。
上游失败为视觉输入设置超时；有缓存则使用缓存，没有则明确走文本模式。
模块耦合用事件总线衔接 ASR、视觉、推理和 TTS，避免模块之间直接互相依赖。

我会如何继续迭代

下一版先补两个能力：生成过程的流式反馈，以及用户说话时能够中止过期任务。前者改善等待体验，后者避免旧回答覆盖新的意图。长期记忆则应在前两项稳定后，再以向量化摘要的形式引入。
