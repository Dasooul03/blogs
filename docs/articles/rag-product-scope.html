RAG · PROJECT SCOPE

一个 RAG 知识库系统，究竟要解决什么？

从本地项目笔记看，RAG 系统不是一个“把问题交给模型”的接口，而是一条把私有资料、外部信息与结果质量控制连接起来的链路。

学习笔记 · 依据本地《项目功能介绍》整理

先明确系统的边界

资料在知识库里时，系统应检索到相关片段，并在答案里保留可追溯的依据。资料不在知识库里时，系统不能把猜测伪装成事实；它要么明确说明资料不足，要么走一条单独标识的外部信息补充路径。这个边界决定了系统的可信度，也决定了用户是否知道答案来自哪里。

三类能力

能力目的要验证的结果
私有知识库问答从已导入资料中找到证据。正确片段能被召回，回答能指向来源。
外部信息补充处理资料没有覆盖的问题。外部内容与本地资料有清楚的来源区分。
结果优化降低无关信息和错误回答。候选被排序、低置信度被暴露，而不是被掩盖。

把需求写成验收标准

“能问答”不是验收标准。更可操作的做法是准备一组问题：一部分答案明确存在于资料中，一部分故意不存在，另一部分需要区分不同主题的文档。对每个问题记录检索来源、最终回答、是否发生臆测，以及用户能否理解系统的不确定性。

功能描述要写成可验证的能力：能回答什么、依据是什么、资料缺失时做什么、系统如何把不确定性展示出来。

下一步练习

从十份结构清晰的 Markdown 开始，先不接联网搜索。等本地问答能够稳定给出来源，再决定是否增加外部信息支路。这样能先建立一条可控的基线，而不是把“检索、搜索、生成”混在一起排错。
