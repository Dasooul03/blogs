LANGGRAPH · STATE

为什么工作流要有显式 State？

当一个文档要经过分流、解析、切片、嵌入和入库，状态不应散落在全局变量、临时文件和函数参数里。

学习笔记 · 依据本地《导入数据图与状态定义》整理

State 是节点之间的契约

每个节点都应说清自己读什么、写什么。入口节点写入文件类型和任务 ID；解析节点写入 Markdown 内容和输出路径；切片节点写入 chunks；嵌入节点写入可持久化的向量结果。这样查看一次任务时，不需要在多个函数里猜“数据现在藏在哪里”。

字段如何分组

字段类别例子为什么需要
追踪任务 ID、错误状态让日志和前端进度对应同一任务。
路由PDF/Markdown 开关决定条件边进入哪个节点。
路径输入、转换、分块和索引位置支持恢复和复查中间产物。
中间结果正文、切片、向量、标签避免节点重复计算或依赖隐式变量。

默认值与测试

State 的默认值也属于设计。布尔开关、空列表和路径字段必须可预测；可变对象要避免任务之间共享引用。测试至少覆盖 PDF、Markdown 和非法格式三条路径，并断言路由开关、输出路径、切片数量和错误状态，而不只是检查流程有没有结束。
